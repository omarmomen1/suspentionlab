# Handoff Report: Alembic Tier Cleanup

## Observation
1. Running `alembic -c src/suspensionlab/backend/alembic.ini check` fails with multiple schema mismatches:
   - Type mismatches on all UUID columns across all tables (detected as changing from `NUMERIC()` to `UUID()`).
   - A leftover temporary table `_alembic_tmp_jobs` exists in the database.
   - Missing/removed indexes on the `jobs` table (`idx_user_active_jobs` and `ix_jobs_user_id`).
2. The initial migration `8f6fb1bdcda6_enterprise_initial.py` created duplicate indexes on `jobs` (`idx_user_active_jobs` and `ix_jobs_user_status` both index `['user_id', 'status']`). The models only define `ix_jobs_user_status`.
3. The migration `d713c8804400_add_collaboration_hub.py` used `batch_alter_table` on `jobs`. In SQLite, this recreates the table and left behind `_alembic_tmp_jobs`.
4. In `src/suspensionlab/backend/services/quota.py` line 51, the quota check uses `tier = user.get("tier", PlanTier.FREE)`. However, the `tier` column was dropped in `8ff4d3e48527_drop_tier_column.py`, and `auth.py` populates the user dictionary with the key `"plan"`. This results in all users being restricted to the `FREE` tier limits.

## Logic Chain
1. **Schema Mismatch (Type Mismatch):** SQLAlchemy 2.0+ maps `UUID` columns to `NUMERIC` or `CHAR(32)` in SQLite. Alembic's autogenerate strictly compares the database's `NUMERIC` type against the model's `UUID` type. Implementing a custom `compare_type` hook in `env.py` to ignore this specific mapping resolves the false positive.
2. **Schema Mismatch (Temp Table & Indexes):** The redundant indexes were removed from the models but remain in the database. A new autogenerated migration script must be created and upgraded to drop `_alembic_tmp_jobs`, `idx_user_active_jobs`, and `ix_jobs_user_id` to fully synchronize the database state with the models.
3. **Tier System Sync:** The database schema correctly uses `plan` for the Free, Pro, and Enterprise tiers. The remaining issue is a logic bug in `quota.py` referencing the old `tier` nomenclature. Changing the dictionary lookup to `"plan"` aligns the application logic with the database schema.

## Caveats
- The custom `compare_type` hook is specifically for SQLite's handling of UUIDs. If the production environment uses PostgreSQL, `UUID` will natively match `UUID` without triggering the hook.
- When running autogenerate, an SQLAlchemy SAWarning may appear (`unresolvable cycles between tables "teams, users"`). This is due to the mutual foreign keys between Teams and Users but is non-fatal.

## Conclusion
To finalize the Alembic database migrations and synchronize the tier system, the implementer should:
1. Update `src/suspensionlab/backend/alembic/env.py` with a custom `compare_type` hook in `run_migrations_online` to ignore `UUID` vs `NUMERIC`/`CHAR` type differences.
2. Run `alembic revision --autogenerate -m "cleanup_schema"` and apply it to drop the leftover `_alembic_tmp_jobs` table and redundant `jobs` indexes.
3. Fix the logic bug in `src/suspensionlab/backend/services/quota.py` (line 51) by changing `user.get("tier")` to `user.get("plan")`.
