SuspensionLab PRO: Distributed Optimization Engine

Bridging the gap between classical theoretical vehicle dynamics and modern, horizontally scaling cloud infrastructure. A production-grade SaaS architecture engineered for high-fidelity ride frequency and damping optimization.

Executive Summary

I architected, developed, and deployed a highly concurrent, cloud-native simulation platform. Moving beyond local, single-threaded MATLAB scripts, I engineered a system that evaluates non-linear state-space models via distributed compute clusters. SuspensionLab PRO enables race engineers to iteratively optimize suspension parameters against complex transient road profiles at scale. I utilized a decoupled Redis/RQ compute plane, protected by strict TOCTOU concurrency constraints, ensuring absolute mathematical fidelity under massive HTTP load.

The Physics Engine

State-Space Integration & Non-Linear Solvers

  • 2-DOF Linear Quarter Car Model with 4-Dimensional State Vector: Developed a quarter-car simulation evaluating sprung (m_s) and unsprung (m_u) mass dynamics. The model state vector [z_s, ż_s, z_u, ż_u] captures displacement and velocity for both masses across aggressive transient profiles.
  • L-BFGS-B Optimization: Implemented the SciPy L-BFGS-B solver to navigate complex, non-linear design spaces, isolating optimal spring rates (Ks) and damping coefficients (Cs). Implemented via SciPy's L-BFGS-B gradient-based solver with bounded line search across the full spring-damper design space, evaluating the complete physics simulation at each iteration (n_freq=600, full road profile duration) to achieve mathematically rigorous convergence within 150–300 function evaluations.
  • Multi-Objective Target Isolation: Engineered composite cost functions targeting specific race outcomes: Comfort (minimizing sprung mass acceleration), Grip (minimizing tire load variation), and a weighted Balanced hybrid.
  • Domain-Specific Failure Routing: Built resilient mathematical boundaries that catch non-convergent matrices and division-by-zero states, explicitly throwing a custom NonRetryableJobError to prevent catastrophic infinite solver loops across the worker nodes.

The SaaS Architecture

I engineered the architecture to guarantee complete isolation between the API routing layer and the heavy computational load. Theoretical physics engines fail in production when they block web threads.

  • Strict Execution Isolation: Decoupled the FastAPI control plane from the SciPy compute plane using a Redis-backed RQ message broker. The HTTP layer remains strictly non-blocking.
  • Event-Driven Kubernetes Autoscaling: Translated the stack into a production Kubernetes deployment. Implemented a KEDA ScaledObject (Horizontal Pod Autoscaler) that monitors the rq:queue:default list length, dynamically scaling the SciPy worker pods from 1 to 10 based entirely on active solver backlog.
  • Contract-Driven Boundaries: Enforced absolute data boundaries between frontend, API, and worker nodes using strict Pydantic schemas, entirely eliminating unverified payload injections.

Site Reliability Engineering (SRE) & Hardening

I aggressively tested and patched the system against massive concurrency, infrastructure death, and malicious mathematical inputs.

  • TOCTOU Race Condition Mitigation: I implemented distributed Redis locks (redis.lock) wrapping critical database transactions. This guarantees atomic evaluation when enforcing per-user concurrent job limits, outright blocking zombie traffic spikes.
  • Orphan Recovery & Idempotency: Engineered a background CronJob sweeper to recover dead lock states if a worker container OOMs. Designed Stripe webhook listeners with database-level primary key constraints to block subscription tier inflation via duplicate network deliveries.
  • Distributed Tracing (OpenTelemetry): Integrated OpenTelemetry across the network partition. I manually rehydrated trace contexts inside the isolated RQ worker processes to build complete, cross-container execution waterfalls from UI click to SciPy completion.
